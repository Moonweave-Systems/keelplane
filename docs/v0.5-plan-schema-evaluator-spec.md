# V0.5 Plan Schema And Evaluator Spec

Status: Implemented, Last updated: 2026-06-14

## Purpose

V0 proved that `keelplane` can describe a large workflow in a
consistent shape. It did not prove that the skill is meaningfully better than a
strong planning prompt, `workflow-router-skill`, the local Claude
`claude-agent-workflow-designer`, or a runtime-backed workflow tool.

V0.5 exists to answer one question before any plugin or runtime work:

> Does the current checked-in skill package define a reusable workflow artifact
> corpus that another agent or future runtime can consume with less
> reinterpretation than nearby alternatives?

If V0.5 cannot show that, stop growing this repo as a separate product and fold
the useful pieces into `workflow-router` or a runtime project.

## Product Boundary

Use a router-first rule. When a prompt can be handled by direct Codex work or
`workflow-router` without losing resumability, handoff, or verification value,
downgrade instead of activating this skill.

This skill is justified only when at least one exclusive condition is true:

- a downstream agent or future runtime is expected to execute the plan later
- the workflow needs resumable handoffs, cached discovery, or restart boundaries
- the workflow needs true multi-surface fan-out across independent repositories,
  packages, artifact types, or evidence sources

After one exclusive condition is met, at least one supporting condition must
also be true:

- the task needs planned fan-out/fan-in across independent surfaces
- the task needs adversarial verification before synthesis
- the task needs explicit human gates for destructive, external, costly, or
  public API work

If no exclusive condition is true, the correct output is a downgrade:

- direct Codex work for small implementation/debugging/review
- `workflow-router` for ordinary broad but bounded routing
- a simple plan when execution does not need multi-agent or resumable structure

The skill must not win by being more verbose. It wins only if its artifact is
more executable, auditable, or reusable.

## V0.5 Deliverables

1. A canonical machine-readable workflow artifact schema.
2. A human-readable render format derived from the same artifact.
3. An evaluator that checks schema validity, activation decisions, and release
   fitness.
4. A fixed benchmark set with positive, negative, and comparative fixtures.
5. A keep/kill gate that decides whether the deterministic sample contract is
   strong enough for the repo to remain separate.

## Canonical Artifact

The output artifact is `workflow.plan.json`. The implemented V0.5 evaluator
validates tracked sample artifacts under `samples/v0.5/`; it does not execute a
live model against `SKILL.md`.

The human-readable blueprint is a render of this JSON. The JSON is the source of
truth. Manual Markdown plans without JSON are not V0.5-complete.

Minimum machine-consumable contract:

- every worker has explicit tool permissions, context budget, and output schema
- every handoff declares a format, required fields, and validation command
- every first slice has instruction, inputs, expected output, completion check,
  and forbidden actions
- every writable or external action is represented as a risk gate, not hidden in
  prose

## Evaluator

Add `scripts/evaluate_plan.py`.

Inputs:

- `--plan <path>` for a single `workflow.plan.json` validation
- `--manifest fixtures/v0.5/manifest.json` for batch evaluation
- `--out <path>` for regenerated evidence; the repo-local path must resolve
  inside this repository

Required checks:

- the manifest declares schema version `0.5`, a non-empty fixture set, and the
  two unique expected baselines: `workflow-router-skill` and
  `claude-agent-workflow-designer`
- `SKILL.md` hash is recorded beside each generated plan.
- `workflow.plan.json` parses and matches the schema.
- `activation.decision` matches the fixture expectation.
- activated plans include one exclusive threshold and one supporting threshold.
- downgrade plans name the downgrade target, use no phases, handoffs, restart
  points, or worker prompt contracts, and route the first slice to the target.
- all `patterns` exist in `references/workflow-patterns.md`.
- every phase has entry criteria, exit criteria, workers, and outputs.
- every worker has typed tool permissions, context budget, and prompt contract.
- every handoff has an artifact schema with required fields and validation
  command.
- handoff phase references, phase dependencies, and resume restart points refer
  to declared phases.
- phase dependencies are acyclic.
- cross-phase handoff targets depend on their source phase.
- resume cacheable outputs are produced by a declared phase.
- every risk gate has a safe default that blocks before the risky action and
  requires user approval.
- risky worker permissions and writable surfaces have corresponding separate
  risk gates using token-level matching, not substring matching.
- every verification item names a falsifier and required evidence.
- `execution_path.first_slice` includes instruction, inputs, expected output,
  completion check, and forbidden actions.
- raw-output wrappers include `raw_kind`, `fixture_id`, prompt, producer,
  current skill hash, packet hashes, parsed plan, and rendered blueprint
  matching the parsed plan.
- consumer reports claim only allowed blinded inputs and match first-slice
  inputs, forbidden actions, risk gates, and safe defaults.
- consumer reports fail closed on unexpected fields and include blinded
  sample-review provenance.
- consumer provenance is recorded as a validated structured record with
  packet hashes for the plan, blueprint, prompt, and repo-surface metadata when
  present, plus field-level support for first slice, inputs, expected output,
  completion check, forbidden actions, risk gates, safe defaults, and downgrade
  target when applicable; it is not a live blinded-review runner, actual checkout
  path hash, or independent derivation proof.
The consumer record still uses the blinded sample-review provenance label.
- baseline normalization-failure records include source hash verification and a
  unique observation-tagged source excerpt plus interpretation for every scored
  observation.
- manifest evaluation verifies that `docs/v0.5-decision.md` matches the freshly
  regenerated summary.

The evaluator must fail closed. Missing, empty, unknown, or contradictory fields
are errors, not warnings.

## Benchmark Fixtures

Store fixtures under `fixtures/v0.5/`.

Positive activation fixtures:

1. API auth audit across a multi-package service.
2. 500-file migration with batching, rollback boundaries, and regression tests.
3. Unsupported-claim audit for a PR description plus repo evidence.
4. Cross-source research synthesis requiring claim verification.

Negative activation fixtures:

1. "Fix this failing unit test."
2. "Review this 40-line diff."
3. "Draft a simple implementation plan for one file."
4. "Explain which workflow pattern this resembles."

Borderline downgrade fixtures:

1. "Plan an auth audit for one service where one agent can inspect all routes."
2. "Create a migration plan for 12 call sites in one package."
3. "Compare two architecture options without downstream execution."

Meta/runtime fixture:

1. "Design a future runtime for this skill without overbuilding."

Each fixture must define:

- expected activation decision
- expected downgrade target when applicable
- required matched thresholds for activation
- required or forbidden patterns
- required risk gates
- linked raw-output and consumer-report records validated by the evaluator

## Comparative Evaluation

V0.5 compares this skill against the two baseline source snapshots named in the
manifest:

- `workflow-router-skill`
- `claude-agent-workflow-designer`

Each baseline fixture record is a tracked, source-hashed normalization-failure
record. V0.5 does not accept tracked normalized baseline plans as evidence.
Normalization-failure records include baseline source hashes, structured
observations, and unique observation-tagged source-backed evidence excerpts;
numeric scores are derived by the evaluator, not hand-authored in the records.
A baseline is not penalized for native format; it is penalized only when the
tracked source-backed observations cannot support the same downstream consumer
task. V0.5 baseline comparison is fixture-indexed, prompt-matched sample record
validation against tracked snapshots, not fresh baseline execution, live
sibling-source comparison, or proof that each observation excerpt is unique to
that fixture prompt.

The baseline evidence contract requires source-backed evidence excerpts.

For the same fixture prompt, score each candidate on a deterministic 0-2 scale.
In V0.5 these are validator-derived gates, not subjective human ratings:
candidate schema-validity and consumer success are binary pass/fail-to-2
signals, while baseline normalization-failure records derive scores from
structured source-backed observations.

| Metric | 0 | 1 | 2 |
| --- | --- | --- | --- |
| Activation discipline | over/under-routes | ambiguous | correct and justified |
| Executable artifact | cannot normalize | normalized with gaps | schema-valid artifact |
| Handoff clarity | needs reinterpretation | usable with questions | agent can start directly |
| Verification strength | generic checks | partial falsifiers | concrete falsifiers and evidence |
| Safety gating | missing gates | partial gates | complete gates with safe defaults |
| Resume value | absent | mentioned | cache/restart rules are actionable |
| Downstream consumer success | consumer cannot start or chooses wrong route | consumer needs clarification | consumer starts the first slice or agrees with downgrade |

The evaluator records scores in `out/v0.5/<fixture>/scorecard.json` with one
numeric field per metric, the normalized artifact path or normalization failure,
and a link to the blinded consumer report when the downstream consumer metric is
scored.

The keep/kill comparison uses these metrics:

- activation discipline
- handoff clarity
- verification strength
- safety gating
- downstream consumer success

Executable artifact score is diagnostic. It does not alone decide the winner.

## Keep/Kill Gate

Keep this repo as a separate skill only if all are true:

- all positive fixtures activate correctly
- all negative fixtures downgrade correctly
- all borderline fixtures downgrade to `workflow-router` unless an exclusive
  threshold is present
- every fixture has a schema-valid artifact or a valid downgrade artifact
- this skill beats both baselines by at least 20 percent aggregate average score
  across activation discipline, handoff clarity, verification strength, safety
  gating, and downstream consumer success
- every fixture has a provenance-bearing blinded downstream consumer report

The evaluator fails fast with a validation error when any fixture fails its
expected outcome, any artifact is schema-invalid, or any contract field drifts.
It emits `kill-or-merge` only after all fixtures validate and the candidate
aggregate keep/kill average does not beat each baseline aggregate by at least 20
percent.

## Downstream Consumer Protocol

The deterministic blinded consumer report is a tracked structured record with
field-level provenance. It must claim only these inputs:

- `workflow.plan.json`
- the rendered blueprint generated from that JSON
- the original fixture prompt
- repository path, when the prompt is repo-bound

The consumer must not receive this spec, the expected answer, scorecard notes,
or author commentary.

Pass criteria:

- the consumer identifies the first slice without asking for missing structure
- the consumer can name the exact inputs it needs
- the consumer can state the expected output and completion check
- the consumer can identify forbidden actions and risk gates before execution
- for downgrade artifacts, the consumer agrees with the downgrade target

Fail criteria:

- the consumer needs human reinterpretation of handoffs, worker roles, or first
  slice
- the consumer starts a risky action without the safe default
- the consumer cannot distinguish execution work from verification work
- the consumer disagrees with a downgrade decision on a borderline fixture

## Architecture

V0.5 has four layers:

1. Skill instructions: define the expected `workflow.plan.json` and rendered
   blueprint contract.
2. Schema/evaluator: validate shape, activation, safety, and handoff quality.
3. Fixture harness: validate fixed prompts and preserve tracked raw outputs plus
   hashes.
4. Comparative report: decide keep, kill, merge, or runtime-track.

This is not a runtime. It does not spawn agents, resume work, watch progress, or
write MCP tools. It creates and validates the artifact corpus a runtime would
later consume. V0.5 does not claim live model generation from `SKILL.md`.

## Release Criteria

V0.5 is releasable when:

- `workflow.plan.json` schema is documented and enforced.
- `scripts/evaluate_plan.py --self-test` passes positive and negative cases.
- exactly 12 fixtures exist: four positive, four negative, three borderline,
  and one meta/runtime.
- manifest-listed candidate, raw, consumer, and baseline record paths exactly
  match the tracked V0.5 sample corpus.
- every fixture has a recorded raw output, parsed plan, current skill hash, and
  score.
- every fixture has a schema-valid artifact or valid downgrade artifact.
- comparative scorecards exist for exactly the two named baseline snapshots.
- baseline records use explicit source-hashed normalization failure with
  source-backed observations.
- a provenance-bearing structured downstream consumer report is validated and
  copied under `out/v0.5/consumer/`.
- keep/kill decision is recorded in `docs/v0.5-decision.md` and matches the
  freshly regenerated `out/v0.5/summary.json`.
- README and `docs/spec.md` point to V0.5 as the continuation gate.
- existing V0 release checks still pass.

## First Implementation Slice

Build the smallest useful slice:

1. Add `references/workflow-plan-schema.md`.
2. Add `fixtures/v0.5/` with one positive and one negative fixture.
3. Add `scripts/evaluate_plan.py` with schema, pattern, activation, and empty
   field checks.
4. Add one hand-written sample `workflow.plan.json` for each initial fixture.
5. Run `evaluate_plan.py --self-test` and fix schema ambiguity before touching
   `SKILL.md`.

Only after that slice passes should `SKILL.md` be updated to require JSON output
for substantial workflow designs.
