## Baselines

The two confirmed baseline snapshots are recorded as fixture-indexed,
prompt-matched records under `samples/v0.5/baselines/`, with source snapshots
under `samples/v0.5/baseline-sources/`:

- `workflow-router-skill`: source-hashed normalization failure because the
  source is a routing policy, not a `workflow.plan.json` producer.
- `claude-agent-workflow-designer`: source-hashed normalization failure because
  the snapshotted source provides prose workflow guidance and scaffolding
  helpers, not per-fixture schema-valid JSON artifacts.

Baseline records do not contain hand-authored numeric scores. The evaluator
derives scores from structured source-snapshot observations, verifies the
tracked source snapshot hash, and requires a unique source-backed evidence
excerpt within each record before using it. V0.5 does not claim each baseline
observation excerpt is unique to that fixture prompt.

## Consumer Result

Deterministic structured consumer records are stored per fixture under
`samples/v0.5/consumer/`; the evaluator validates and copies them under
`out/v0.5/consumer/`.

Result: pass. Each consumer report claims only the allowed blinded inputs,
identifies the exact first-slice inputs, expected output, completion check,
forbidden actions, risk gates, and safe defaults, records blinded sample-review
provenance, and does not receive this spec or expected-answer notes. Downgrade
fixtures also agree with the downgrade target.

The consumer evidence format is a validated structured record. It requires
packet hashes for the plan, blueprint, prompt, and repo-surface metadata when
present, plus blinded sample-review provenance with field-level support for first
slice, inputs, expected output, completion check, forbidden action, risk gate,
safe default, and downgrade target fields, but it is not a live blinded-review
runner, actual checkout path hash, or independent derivation proof.

## Raw Artifacts

Tracked raw outputs are stored under `samples/v0.5/raw/`. The evaluator rejects
raw output that merely duplicates the parsed plan; each raw record wraps the
prompt, current `SKILL.md` hash, rendered blueprint, and parsed `workflow_plan`.
The evaluator rejects prompt mismatch, stale skill hash, or blueprint drift before
regenerating `out/v0.5/<fixture>/raw-output.json`.

## Boundary

V0.5 does not claim runtime execution or live model generation from `SKILL.md`.
It also does not claim fresh baseline execution or live sibling-baseline
comparison. It proves that the tracked fixture corpus is bound to the current skill hash,
can be serialized as `workflow.plan.json`, rendered as a blueprint,
and evaluated deterministically before a future plugin or runtime slice. The
manifest gate also verifies this decision document against the freshly
regenerated `out/v0.5/summary.json`.
