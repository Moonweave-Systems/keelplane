# V5.5 Worker Result Review Spec

Status: planned
Date: 2026-06-15

## Purpose

V5.5 reviews V5 worker-result evidence before any runtime treats a phase as
complete. V5 proves a fixture worker produced output. V5.5 decides whether that
output satisfies the scheduled packet contract.

This slice closes the trust gap:

```text
V4 scheduled packet
-> V4.5 dispatch bundle
-> V5 worker result evidence
-> V5.5 reviewed worker result
```

The output is still not workflow advancement. It is a reviewed result artifact
that a later runtime-advance slice can consume.

## Product Position

V5.5 is the worker-result counterpart to V2.5. V2.5 reviews a first-slice
execution attempt. V5.5 reviews a later scheduled worker result.

The review contract must be deterministic, hash-bound, and resumable. A worker
result is not trusted because the fixture command exited successfully; it is
trusted only when the review can tie produced outputs back to the original V4
packet, dispatch bundle, and V5 result hashes.

## Command Contract

```bash
python scripts/review_worker_result.py --result out/v5/<run_id> --out out/v5.5/<run_id>
python scripts/review_worker_result.py --resume out/v5.5/<run_id>
python scripts/review_worker_result.py --self-test
```

## Accepted Inputs

V5.5 accepts only an owned V5 result directory with:

- `.run_worker_result-owned.json`,
- `status.json` with `status: executed`,
- `result.json`,
- `hashes.json`,
- `stdout.txt`,
- `stderr.txt`,
- produced outputs under `work/`,
- source dispatch hashes that still validate through V4.5 resume.

V5.5 rejects:

- missing or symlinked files,
- stale V4.5 dispatch source,
- tampered V5 result, hashes, stdout, stderr, or produced outputs,
- `failed`, `blocked`, or `invalid` V5 statuses,
- results with no produced outputs,
- outputs outside owned `out/v5/<run_id>/work/`.

## Output Model

Accepted review writes:

```text
out/v5.5/<run_id>/
├── .review_worker_result-owned.json
├── review.json
├── review.md
├── hashes.json
├── status.json
└── resume.md
```

`review.json` contains:

- `review_id`,
- `verdict`,
- `source_result_path`,
- `source_packet_id`,
- `source_phase_id`,
- `source_hashes`,
- `findings`,
- `approved_outputs`.

## Verdicts

- `approve`: output satisfies the packet's expected output contract.
- `request-changes`: output exists but does not satisfy expected output checks.
- `needs-human`: deterministic review cannot decide safely.
- `invalid`: source evidence or review artifacts are malformed or stale.

The first fixture review may approve only `verification.md` for the
`evidence_review` dogfood packet. It should not generalize to arbitrary output
schemas yet.

## Review Rules

For the first slice, `approve` requires:

1. V5 status is `executed`.
2. V5 resume is `resumable`.
3. `result.json.phase_id` equals `evidence_review`.
4. `verification.md` is listed in `produced_outputs`.
5. `work/verification.md` exists and is non-empty.
6. The output hash matches V5 `hashes.json`.
7. The source packet expected outputs include `verification.md`.
8. The source packet stop conditions do not allow direct V5 runtime advance.

Any hash or ownership failure is `invalid`, not `request-changes`.

## Resume Behavior

`--resume` recomputes the source V5 result, source hashes, review body, markdown,
and review hashes. It writes only `status.json` and `resume.md`.

Resume returns:

- `review-approved` with `resume_state: resumable` when everything matches,
- `changes-requested` when deterministic output checks fail,
- `needs-human` when the source is valid but outside the fixture review scope,
- `invalid` when source or review artifacts are stale or malformed.

## Non-Goals

- Do not advance V3/V4 runtime.
- Do not execute workers.
- Do not repair worker outputs.
- Do not merge outputs into the repository.
- Do not interpret arbitrary output schemas.
- Do not call Codex CLI, OMX, network APIs, paid APIs, or external services.
- Do not commit, push, install dependencies, deploy, access secrets, delete
  files, rewrite history, or send external messages.

## Fixtures

The first slice must cover:

- approve `out/v5/v32-semantic-dogfood`,
- reject missing `verification.md`,
- invalidate tampered output hash,
- invalidate tampered review artifact,
- clean resume,
- stale source result invalidation,
- needs-human for unsupported phase or output schema.

## Release Criteria

The slice is `keep` only if:

- `python scripts/review_worker_result.py --self-test` passes,
- dogfood review over `out/v5/v32-semantic-dogfood` returns
  `review-approved`,
- clean resume is resumable,
- tampered review and tampered source evidence invalidate resume,
- no runtime advancement or worker execution is introduced.

## Next Slice

After V5.5, the next slice should be V6 runtime ingestion: consume reviewed V5.5
results as completed phase evidence and generate the next scheduler frontier.
That slice must still reject unreviewed V5 results.
